# ADR-007: AI Harness SDD

## Estado

Aceptado

## Contexto

El framework debe guiar a agentes IA como opencode para seguir el workflow SDD de forma estricta:
- No implementar hasta que specs estén aprobadas
- Tests antes de código
- Documentación como fuente de verdad

## Decisión

Módulo `harness` integrado en `domain` que:

### 1. StageEnforcer

Verifica que el gate de etapa esté pasado antes de permitir implementación.

```java
public interface StageEnforcer {
    void enforceCurrentStage();
    boolean canImplement(ActivityId activityId);
    Stage getCurrentStage();
}
```

### 2. DocumentationVerifier

Confirma que la documentación de la actividad está en estado `accepted`.

```java
public interface DocumentationVerifier {
    boolean isActivityApproved(ActivityId activityId);
    boolean hasAllCriteria(ActivityId activityId);
}
```

### 3. TestFirstEnforcer

Asegura que los tests existan antes de implementar código de producción.

```java
public interface TestFirstEnforcer {
    boolean hasTestsFor(ActivityId activityId);
    void enforceTestFirst(ActivityId activityId);
}
```

### 4. StatusTracker

Compara estado de documentación vs implementación.

```java
public interface StatusTracker {
    ProjectStatus getStatus();
    void updateActivityStatus(ActivityId id, Status status);
    void markTaskComplete(TaskId id);
}
```

### 5. OpencodeIntegration

Interface para que opencode consulte el estado del proyecto.

```java
public interface OpencodeIntegration {
    boolean canStartImplementation(ActivityId activityId);
    List<String> getBlockingReasons(ActivityId activityId);
    String getStatusReport();
}
```

## Integración con opencode

El agente opencode consulta `status.yml` antes de cualquier tarea de escritura:
- Si actividad no está `accepted`, no implementar
- Si criteria no están completos, no implementar
- Si tests no existen para la actividad, no generar código de producción

## Flujo de Verificación

```
Agente quiere trabajar en actividad
    ↓
Consultar status.yml
    ↓
¿Actividad tiene status = accepted?
    ├── NO → Alertar: "Actividad no aprobada. No implementar."
    ↓
¿Todos los criterios de aceptación tienen tests?
    ├── NO → Alertar: "Primero escribir tests."
    ↓
¿Gate de etapa pasado?
    ├── NO → Alertar: "Gate no pasado. Completar requisitos."
    ↓
Implementar
```

## Consecuencias

- Agente es bloqueado si intenta implementar sin aprobación
- `bbook plan` muestra alertas de qué está bloqueando
- Documentación es fuente de verdad
- Stages deben pasar en orden: Definition → Design → Implementation → Deployment
