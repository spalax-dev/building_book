# ADR-006: Arquitectura por Capas

## Estado

Aceptado

## Contexto

Necesitamos una arquitectura que permita:
- Separación clara de responsabilidades
- Testabilidad de la lógica de negocio
- Flexibilidad para cambiar implementaciones de infraestructura
- Integración con opencode para agentes IA

## Decisión

Arquitectura por capas con inyección de dependencias:

```
Infraestructura → Dominio ← Presentación
```

### Presentación (Presentation)

Capa de comunicación con el exterior:
- CLI (Picoli/GraalVM): Comandos `bbook build`, `bbook open`, `bbook plan`
- Web (Pebble templates): Servidor HTTP con live reload
- API: Rutas REST

### Dominio (Domain)

Lógica de negocio pura:
- Solo Java, sin dependencias de frameworks
- Interfaces y abstracciones
- Sin estado de infraestructura
- Testeable sin mocks

### Infraestructura (Infrastructure)

Comunicación con sistemas externos:
- PlantUML: Renderizado de diagramas
- YAML: Lectura/escritura de configuración
- HTTP: Cliente para descargas
- FileSystem: Lectura/escritura de archivos

## Consecuencias

- Dominio no conoce implementación de infraestructura
- Presentación depende de dominio, no de infraestructura
- Infraestructura implementa interfaces definidas en dominio
- DI conecta las capas en `Main` o módulo de bootstrap

## Estructura de Paquetes

```
com.buildingbook
├── presentation
│   ├── cli        # Comandos Picoli
│   ├── web        # HTTP Server + Pebble
│   └── api        # REST endpoints
├── domain
│   ├── model      # Entidades (Document, Process, Activity, Task)
│   ├── service    # Lógica de negocio (StageEnforcer, StatusTracker)
│   └── port       # Interfaces (DocumentRepository, DiagramRenderer)
└── infrastructure
    ├── persistence  # YAML file operations
    ├── plantuml    # PlantUML integration
    └── http        # HTTP client
```
